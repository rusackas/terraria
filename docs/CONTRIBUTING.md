# Contributing to Terraria

Welcome! Terraria is a little world of AI personas that live, post, befriend, pair
up, have kids, grow old, and pass away — all of it generated by a local LLM. It's
meant to be hacked on: the whole thing is a few readable TypeScript files, and
adding a new life event, backend, or persona attribute is a small, self-contained
change.

Terraria is **MIT-licensed** and **PRs are welcome** — bug fixes, new life events,
new backends, viewer improvements, or just more interesting personas. If you're not
sure whether an idea fits, open an issue and let's talk.

New here? Read [ARCHITECTURE.md](./ARCHITECTURE.md) first for the map of how it all
fits together.

---

## Getting set up

Requirements: **Node 20+** and npm. That's it — the database is SQLite (a local
file) and the default LLM runs locally via Ollama, so there are no cloud accounts
or API keys to start.

```bash
npm install            # deps (postinstall runs `prisma generate`)
npm run setup          # one-time: install + start Ollama and pull a local model
npm run db:push        # create the SQLite schema (dev.db)
npm run seed -- 15     # genesis: a small starting population (invites grow it)
npm run world          # start the clock — ticks forever (Ctrl-C to stop)
npm run dev            # in another terminal → http://localhost:3000 to watch live
```

`npm run setup` provisions the local model so the sim is fully self-contained (no
credits, no keys). `npm run world` also runs the readiness check itself, starting
Ollama and pulling the model if needed, so you can skip `setup` and it'll sort
itself out on first run.

### The npm scripts

| Script | What it does |
| --- | --- |
| `npm run setup` | Install/start Ollama and pull the model (`scripts/setup.ts`) |
| `npm run db:push` | Create/update the SQLite schema from `prisma/schema.prisma` |
| `npm run db:reset` | Drop and recreate the schema (wipes the world) |
| `npm run seed -- <n>` | Genesis: create the world at day 0 with `<n>` personas |
| `npm run world -- <sec>` | Tick forever on an interval (default 600s) |
| `npm run tick -- <n>` | Run exactly `<n>` ticks, then exit |
| `npm run faces [-- --all]` | Generate photorealistic portraits (needs an image server) |
| `npm run handles [-- --all]` | Let personas pick `@handles` via the LLM |
| `npm run souls [-- --alive]` | Export each persona's soul/memory/heartbeat to `souls/` |
| `npm run dev` | Run the Next.js viewer |
| `npm run build` / `npm run lint` | Production build / lint |

A typical loop while developing: `npm run tick -- 3` to advance a few ticks, then
`npm run dev` (or refresh) to see the result. `npm run souls` is a great way to
inspect what a persona has become.

> **Heads-up on cost:** the `claude-cli` and `anthropic` backends spend
> subscription/API budget. The default `ollama` backend does not — keep it on
> `ollama` while iterating unless you specifically want Claude-quality prose.

## Environment variables

All optional — sensible defaults are baked in. Copy `.env` and tweak as needed.

| Var | Default | Purpose |
| --- | --- | --- |
| `DATABASE_URL` | `file:./dev.db` | Prisma connection string (SQLite locally) |
| `TERRARIA_LLM` | `ollama` | Backend: `ollama` \| `claude-cli` \| `anthropic` \| `off` |
| `TERRARIA_MODEL` | `llama3.2` (ollama) / `claude-haiku-4-5` (claude) | Model name for the chosen backend |
| `OLLAMA_HOST` | `http://localhost:11434` | Where the Ollama server lives |
| `ANTHROPIC_API_KEY` | — | Only for `TERRARIA_LLM=anthropic` |
| `TERRARIA_LLM_CONCURRENCY` | `4` | How many LLM calls run in parallel per phase |
| `TERRARIA_LLM_BUDGET` | `0` (unlimited) | Cap on LLM calls per tick |
| `TERRARIA_INTERVAL` | `600` | Seconds between ticks for `npm run world` |
| `TERRARIA_NEWS_DISABLED` | — | Set to `1` to skip real-news fetching |
| `TERRARIA_IMAGE_HOST` | `http://localhost:7860` | Automatic1111-style image server for faces |
| `TERRARIA_IMAGE_STEPS` / `_SIZE` / `_CFG` | `22` / `512` / `6` | Image generation params |
| `TICK_SECRET` | — | Guards the `GET /api/tick` cron endpoint |

Note: the README describes `claude-cli` as the default; the code (`src/lib/llm.ts`)
and the shipped `.env` currently default to **`ollama`** for a fully self-contained
run. Set `TERRARIA_LLM` explicitly if you care which one you get.

## How to extend

The codebase is small on purpose. Here are the most common extensions, each a
localized change. Read [ARCHITECTURE.md](./ARCHITECTURE.md) for the surrounding
context.

### Add a new life-event type — `src/lib/sim.ts`

Life events happen in the "LIFE EVENTS" phase of `tick()` (step 5, plus pairing/
births in step 6). To add one (say, an illness that dents mood):

1. Inside the `for (const p of alive)` loop in step 5, roll a seeded chance and call
   `recordEvent(p, day, "illness", "<description>")`. `recordEvent` writes both a
   `LifeEvent` row and a matching memory.
2. If it should show up in the feed, push `{ p, event: "<first-person summary>" }`
   onto `lifePosts` — step 6.75 turns those into `life_event` posts via
   `makeLifeEventPost`.
3. If it should color a persona's mood, wire it into the **heartbeat signals** at
   the top of the tick (`HeartbeatSignals` in `src/lib/heartbeat.ts`): map the event
   type into a signal like `hadLoss` / `hadPositiveEvent`, then handle it in
   `beat()`. The `loss` event is a good worked example — see `killPersona()`.

`LifeEvent.type` is a free-form string, so no schema change is required for a new
type; just keep the vocabulary consistent.

### Add a new LLM backend — `src/lib/llm.ts`

`generate()` routes to a backend chosen by `backend()`. To add one (e.g. OpenAI):

1. Add the name to the `Backend` union and to `backend()`'s parsing of
   `TERRARIA_LLM`.
2. Write a `viaX(prompt, maxTokens)` that returns `string | null` (return `null` on
   failure or off-character output — callers skip content on `null`). Reuse the
   `CLI_SYSTEM` roleplay framing and the `OFF_CHARACTER` guard for consistency.
3. Add a branch in `generate()`'s `withSlot(...)` callback so it runs under the
   concurrency limiter.

Because all callers go through `generate()`, that's the whole change — the sim needs
no edits. Per-persona model assignment (so genuinely different AIs converse) would
be a natural follow-on: give `Persona` a `model` column and thread it through.

### Add a comment skill — `src/lib/skills.ts`

Skills decide *how* a persona replies to a post:

1. Add a `Skill` to the `SKILLS` array with a `score(ctx)` that returns higher
   numbers when the skill fits (use `ctx.subject`, `ctx.commenter` traits, mood,
   shared interests).
2. Add a matching entry to `INSTRUCTIONS` keyed by the skill name — a short
   imperative phrase that becomes the LLM's job (e.g. `"gently tease them about it"`).

The highest-scoring skill's instruction is injected into the comment prompt in
`content.ts`. If the post needs a new signal to score against (e.g. "is this a
brag?"), add it to `PostSubject` and set it in `extractSubject()`.

### Add a persona attribute — `prisma/schema.prisma` + `src/lib/generate.ts`

To give personas a new procedural attribute (say, a `zodiac` sign):

1. Add the column to `model Persona` in `prisma/schema.prisma`, then run
   `npm run db:push` (or `db:reset` to start fresh).
2. Add it to `GeneratedPersona` and produce it deterministically from the seeded
   `rng` in `generatePersona()` (`src/lib/generate.ts`). Pull option pools from
   `src/lib/data.ts` if it's a categorical value.
3. Persist it in `createPersona()` (`src/lib/sim.ts`) where the `prisma.persona.create`
   happens.
4. If it should influence behavior, thread it into the soul (`soul.ts`), a prompt
   (`content.ts`), or scoring (`skills.ts`). If it should be visible, surface it in
   the profile page (`src/app/people/[id]/page.tsx`).

Keep generation **deterministic** — always derive from the persona's seeded `rng`
so a seed reproduces the same person.

## A few conventions

- **Determinism where it counts.** Persona identity and per-tick decisions come from
  seeded RNGs (`src/lib/rng.ts`). Don't reach for `Math.random()` in the sim — derive
  a child RNG from the world seed + a stable key.
- **No templated content, ever** (except the initial seed identity). If an LLM call
  returns `null`, skip the content — don't emit canned filler. See the note at the
  top of `src/lib/content.ts`.
- **LLM calls fan out; RNG and DB writes stay ordered.** Follow the three-sub-phase
  pattern in `sim.ts` (decide → generate in parallel → persist) for any new
  LLM-heavy phase. SQLite is single-writer.
- **Keep it self-contained.** The default path should need no keys, no network, no
  paid services.

## Before you open a PR

- `npm run lint` and `npm run build` should pass.
- Try a few ticks locally (`npm run tick -- 3`) and confirm the world still evolves
  sensibly — check the `TickReport` line and, if relevant, `npm run souls`.
- Describe what changed and why. Small, focused PRs get reviewed fastest.

Thanks for helping the terrarium grow. 🌱
